# 设计模式对比与选择

## 创建型模式对比

| 模式 | 使用场景 | 优点 | 缺点 |
|------|---------|------|------|
| 单例 | 全局唯一实例 | 控制实例数量 | 违反单一职责 |
| 工厂 | 对象创建逻辑复杂 | 解耦创建和使用 | 增加类数量 |
| 建造者 | 复杂对象构建 | 链式调用优雅 | 产生多余对象 |
| 原型 | 对象创建成本高 | 性能优良 | 深拷贝复杂 |

## 结构型模式对比

| 模式 | 使用场景 | 优点 | 缺点 |
|------|---------|------|------|
| 适配器 | 接口不兼容 | 提高复用性 | 增加复杂度 |
| 装饰器 | 动态添加功能 | 比继承灵活 | 产生小对象 |
| 代理 | 控制访问 | 职责清晰 | 可能影响性能 |
| 外观 | 简化复杂系统 | 降低依赖 | 不符合开闭原则 |

## 行为型模式对比

| 模式 | 使用场景 | 优点 | 缺点 |
|------|---------|------|------|
| 策略 | 算法可替换 | 避免条件判断 | 策略类增多 |
| 观察者 | 一对多通知 | 降低耦合 | 通知耗时 |
| 责任链 | 多对象处理请求 | 灵活性高 | 不保证处理 |
| 模板方法 | 算法骨架固定 | 代码复用 | 类数量增加 |
| 命令 | 请求封装 | 支持撤销 | 命令类增多 |

## 相似模式对比

### 代理 vs 装饰器
- **代理模式**：控制对象访问，不改变接口
- **装饰器模式**：增强对象功能，可以叠加

### 策略 vs 模板方法
- **策略模式**：整个算法可替换，使用组合
- **模板方法**：算法骨架固定，使用继承

### 适配器 vs 外观
- **适配器**：转换接口，一对一
- **外观**：简化接口，一对多

### 工厂 vs 建造者
- **工厂模式**：创建简单对象，一步完成
- **建造者**：创建复杂对象，多步构建

## 模式组合使用

### 常见组合
1. **单例 + 工厂**：全局唯一的工厂实例
2. **策略 + 工厂**：工厂创建不同策略
3. **装饰器 + 责任链**：中间件实现
4. **观察者 + 单例**：全局事件总线
5. **命令 + 责任链**：请求处理管道

### 实际案例

#### Web 中间件
```
责任链 + 装饰器 + 命令
```

#### 日志系统
```
单例 + 装饰器 + 策略
```

#### 数据处理管道
```
责任链 + 模板方法 + 策略
```

## 选择决策树

### 创建对象时
```
需要全局唯一？
  是 → 单例模式
  否 → 创建逻辑复杂？
    是 → 对象有多个部分？
      是 → 建造者模式
      否 → 工厂模式
    否 → 克隆现有对象？
      是 → 原型模式
      否 → 直接创建
```

### 组合对象时
```
接口不兼容？
  是 → 适配器模式
  否 → 需要简化接口？
    是 → 外观模式
    否 → 需要动态添加功能？
      是 → 装饰器模式
      否 → 需要控制访问？
        是 → 代理模式
```

### 对象交互时
```
需要算法可替换？
  是 → 策略模式
  否 → 需要通知多个对象？
    是 → 观察者模式
    否 → 需要多个对象处理请求？
      是 → 责任链模式
      否 → 需要封装请求？
        是 → 命令模式
        否 → 算法骨架固定？
          是 → 模板方法模式
```

## Go 语言特有考虑

### 使用接口而非继承
Go 的接口是隐式实现的，更适合：
- 策略模式
- 适配器模式
- 装饰器模式

### 使用组合而非继承
Go 推荐组合，更适合：
- 装饰器模式
- 代理模式
- 外观模式

### 使用 Channel 和 Goroutine
Go 的并发特性适合：
- 观察者模式（发布订阅）
- 生产者消费者模式

### 使用函数式编程
Go 支持函数作为一等公民：
- 策略模式（函数作为策略）
- 命令模式（函数作为命令）
- 模板方法（回调函数）
